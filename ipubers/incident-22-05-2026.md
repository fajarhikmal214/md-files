# INCIDENT REPORT
## Database Performance Degradation - Ipubers Production Environment

**Report Date:** 22 May 2026  
**Incident Time:** 22 May 2026 - 09:50 WIB  
**Status:** 🟢 Resolved  
**Classification:** Database Performance - Query Lock Contention

---

# Executive Summary

Pada pukul **09:50 WIB**, database production Ipubers mengalami lonjakan CPU hingga hampir **100%** akibat **query lock contention** yang menyebabkan blocking antar query. Dampak utama terlihat pada layanan **API Integrasi Kartan** yang mengalami timeout dan peningkatan error rate, sementara **API Core Ipubers** tetap berjalan normal.

### Key Impact
- CPU Database mencapai ~100%
- Durasi insiden sekitar 5 menit
- Connection pool mengalami saturasi
- Query waiting meningkat drastis
- 1 service terdampak langsung

# Root Cause Analysis

## Primary Root Cause

Insiden disebabkan oleh **query lock contention** yang memicu blocking berkepanjangan pada database.

### Chain of Events
1. Long-running query menahan exclusive lock
2. Query lain masuk ke waiting queue
3. Connection pool penuh oleh query waiting
4. CPU meningkat akibat lock management & context switching
5. Service dependent mengalami timeout

---

## Contributing Factors

### 1. Missing / Inefficient Index
- Full table scan meningkatkan lock duration
- Lock escalation lebih mudah terjadi

### 2. Inefficient Query Plan
- Query berjalan lebih lama dari normal
- Join dan filtering tidak optimal

### 3. Connection Pool Saturation
- Waiting query tetap memegang koneksi
- Pool cepat habis

### 4. Traffic Spike
- Lonjakan request mempercepat penumpukan query

---

# Preventive Action

### Short Term
- Optimasi query bermasalah
- Tambah/memperbaiki index
- Atur query timeout lebih ketat
- Monitoring blocking query realtime

### Long Term
- Query performance review berkala
- Audit execution plan
- Implementasi alert CPU & blocking threshold
- Capacity planning connection pool

---

# Conclusion

Insiden terjadi akibat query lock contention yang menyebabkan blocking cascade, connection pool exhaustion, dan lonjakan CPU database hingga ~100%.

Dampak utama terjadi pada API Integrasi Kartan karena menggunakan cancellation token, sementara API Core Ipubers tetap stabil karena tidak bergantung pada pembatasan waktu menggunakan cancellation token.

Setelah blocking query diatasi dan resource database kembali normal, seluruh service dapat pulih tanpa downtime berkepanjangan.
